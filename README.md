## 1. 项目名称
**C++⾼性能线程池框架**

---

## 2. 项目介绍

本项目实现了一个基于 C++20 的高性能线程池框架，目标是构建一个**可扩展、可观测、具备工程可用性**的通用并发执行组件。

相比传统线程池实现，本项目重点解决：

- 线程频繁创建销毁带来的性能损耗
- 高并发下任务调度不均衡
- 阻塞任务导致线程池"卡死"
- 缺乏任务执行状态与性能监控能力
- 线程池与任务系统强耦合，难以扩展

为此我设计了：
- 任务队列与线程池解耦架构
- 双模式线程池（FIXED / CACHED）
- 任务执行超时检测机制
- 全链路性能监控系统

该项目可作为：
- 通用线程池组件
- 高并发服务基础模块
- 后续分布式调度系统的基础

## 3. 核心特性

### 线程池 & 队列解耦

该设计本质上将任务生产与任务执行解耦，形成生产者-多消费者模型，提高系统吞吐能力与扩展性。
- 线程池关闭后不主动解绑任务队列
- 重启后可继续消费历史任务
- 仅在线程池销毁时解除绑定关系
  
### 双模式工作支持

- **FIXED模式**：稳定负载-固定线程数，（默认）核心线程数=最大线程数，适用于任务量稳定的场景，避免线程频繁创建销毁开销；
- **CACHED 模式**：突发负载-动态扩缩容，支持根据任务量自动调整线程数（核心线程数~最大线程数），适用于任务量波动大的场景，提升资源利用率。

### 优先级队列调度

- 支持 **MaxHeap（HIGH优先）** 和 **MinHeap（LOW优先）** 两种调度策略；
- 运行时可动态切换优先级模式，自动重建堆结构；
- 确保高优先级任务优先被调度执行，满足业务优先级需求。

### 任务执行超时检测

- 实现**安全软超时机制**，不强行终止线程，避免未定义行为；
- 任务执行完成后判断耗时，通过 `TaskResult.isTimeout` 标记超时状态；
- 调用者可通过 `future.get()` 明确感知任务是否超时；
- 超时任务统一计数统计，便于监控与排错。

### 任务执行性能统计

- 基于 **CAS 无锁原子操作** 实现高性能统计；
- 统计指标包括：总执行任务数、异常失败任务数、超时任务数、所有任务总耗时、单任务最大耗时、平均任务耗时；
- 提供对外查询接口，支持上层业务获取性能数据。

### 内部接口访问控制机制

- 引入 **ThreadPoolPrivateKey 钥匙类** 设计；
- 利用 `friend` 窄授权 + 私有构造，实现接口访问权限隔离；
- 队列的管控接口（如 `taskTake`、`recordTaskFinish` 等）仅线程池可调用，外部主线程无法直接操作，保证架构安全。
  
### 线程池稳定性保障

为保证线程池稳定性：
- FIXED 模式内置守护者线程，实时监控线程状态，自动补充异常退出的线程，保证线程数稳定；
- CACHED 模式内置管理者线程，定期清理空闲线程、动态调整线程数，优化资源占用；

### 完善的任务管理

- 支持任意类型任务（有/无返回值），通过 std::any 实现类型擦除，统一任务接口；
- 任务队列支持容量限制，队列满时自动拒绝新任务，避免内存溢出；
- 基于 std::promise/std::future 实现任务结果传递和异常捕获，确保异常可追溯。

### 优雅关机机制

- 关闭时唤醒所有线程处理剩余任务，拒绝新任务，确保资源安全回收。

### 高可配置性

- 支持动态调整任务队列容量、CACHED 模式扩容步长，支持线程池重命名、重启等操作，适配多样化需求。

### 线程安全设计

- 采用原子变量、互斥锁、条件变量等同步机制，确保多线程环境下任务调度、状态管理的线程安全。

- 依赖：无第三方依赖，仅依赖 C++ 标准库。

1. 项目文件已拆分为：
   - `ThreadPool.h`：核心声明、模板、结构体
   - `ThreadPool.cpp`：非模板实现
   - `main.cpp`：测试文件

```bash
g++ main.cpp ThreadPool.cpp -o threadpool_test -std=c++20 -lpthread
```
```cmd
g++ test.cpp ThreadPool.cpp -o threadpool_test -std=c++20    
```
3. 运行
```bash
./threadpool_test    # Linux/macOS

threadpool_test.exe  # Windows
# 若乱码，输入下方命令后重新启动

```

### 使用示例

```cpp
#include "ThreadPool.h"
#include <iostream>
#include <future>

int main() {
    // 1. 创建任务队列（容量限制为 100，默认 HIGH 优先）
    TaskQueue queue(100, TaskQueue::MaxHeap);
    
    // 2. 创建 FIXED 模式线程池
    // 参数：线程池名称, 绑定的任务队列, 模式, 核心线程数（默认为cpu线程数）
    ThreadPool pool("Test-Pool", &queue, PoolMode::FIXED, 2);
    
    // 2. 或创建 CACHED 模式线程池
    // 参数：线程池名称, 绑定的任务队列, 模式, 最小线程数，最大线程数
    // ThreadPool pool("Test-Pool", &queue, PoolMode::CACHED, 2, 12);

    // 3. 提交无返回值任务（默认 LOW 优先级，不超时）
    auto f1 = pool.submitTask([]() {
        std::this_thread::sleep_for(std::chrono::milliseconds(200));
        std::cout << "无返回值任务执行完成" << std::endl;
    });
    f1.get();  // 等待任务完成
    
    // 4. 提交有返回值任务（显式指定 HIGH 优先级，超时限制 1000ms）
    auto f2 = pool.submitTask(
        Task::PriorityLevel::HIGH,
        std::chrono::milliseconds(1000),
        [](int a, int b) {
            return a + b;
        },
        10, 20
    );

    TaskResult res = f2.get();
    if (res.isTimeout) {
        std::cout << "任务执行超时！" << std::endl;
    } else {
        int sum = std::any_cast<int>(res.value);
        std::cout << "10 + 20 = " << sum << std::endl;
    }
    
    // 5. 动态切换优先级模式（切换为 LOW 优先）
    pool.changeMode(TaskQueue::MinHeap);
    
    // 6. 查询性能监控数据
    std::cout << "总任务数：" << queue.getTotalTaskCount() << std::endl;
    std::cout << "平均耗时(us)：" << queue.getAvgCostUs() << std::endl;
    
    // 7. 手动关闭线程池（析构自动调用）
    pool.shutdownPool();
    
    // 8. 重命名 (仅线程池关闭时可调用)
    pool.Rename("Test-Pool-1");

    // 9. 重启线程池可接着执行任务
    pool.resumePool();
    
    // 10. 模拟高并发请求处理 0ms表示不设置超时时长
    for (int i = 0; i < 1000; ++i) {
        pool.submitTask(Task::PriorityLevel::HIGH, 0ms, [](int id) {
            // 模拟业务处理
            std::this_thread::sleep_for(std::chrono::milliseconds(10));
            return id * 2;
        }, i);
    }
    
    return 0;
}
```

## 6. 详细说明

### 核心模块

#### 任务 (Task)

任务封装结构体：统一管理任意类型任务（支持有 / 无返回值、异常传递、超时限制）
- 用可调用对象封装器打包
  `std::function<std::any()> func;` 将任务参数打包进 func 内，无参数，返回 any（兼容任意返回值）
- 智能指针管理 promise
  `std::shared_ptr<std::promise<TaskResult>> promise;` 用于传递任务结果 / 异常 / 超时状态
- 超时限制
  `Ms m_timeout{0};` 任务执行超时时间，0 表示不限制

#### 任务队列 (TaskQueue)

任务队列是线程池的任务载体，负责存储、管理任务，并支持多线程池绑定，实现添加任务多线程池竞争处理。核心接口：
- `submitTask(Task::PriorityLevel level, Ms timeout, Func&& func, Args&&... args)`：提交任务（支持优先级、超时），返回 `std::future<TaskResult>`；
- `shutdownQueue()`：关闭队列，拒绝新任务，唤醒所有绑定线程池的线程处理剩余任务；
- `setMaxCapacity(int capacity)`：动态调整队列最大容量；
- `changeMode(PriorityMode mode)`：动态切换优先级调度模式；
- `taskTake(ThreadPoolPrivateKey)`：工作线程调用（仅线程池可调用），从队列取出任务；
- `bindConditionVariable(std::condition_variable* cond)`：绑定线程池的条件变量，实现任务入队时唤醒线程；
- `unbindConditionVariable(std::condition_variable* cond)`：线程池销毁 / 关机时调用，避免野指针；
- 性能监控查询接口：`getTotalTaskCount()`、`getFailTaskCount()`、`getTimeoutTaskCount()`、`getMaxCostUs()`、`getAvgCostUs()`。

#### 线程池（ThreadPool）

线程池是核心调度模块，负责管理工作线程、调度任务执行。

**构造函数：**
- FIXED 模式：`ThreadPool(std::string name, TaskQueue* taskq, PoolMode::FIXED, int livenum = std::thread::hardware_concurrency());`
- CACHED 模式：`ThreadPool(std::string name, TaskQueue* taskq, PoolMode::CACHED, int min, int max);`

**核心功能接口：**
- `submitTask(Func&& func, Args&&... args)`：提交任务（默认 LOW 优先级，不超时）；
- `submitTask(Task::PriorityLevel level, Ms timeout, Func&& func, Args&&... args)`：提交任务（显式指定优先级和超时）；
- `shutdownPool()`：关闭线程池，回收所有线程；
- `resumePool()`：重启已关闭的线程池（队列未关闭时有效）；
- `setEverytimeAddCount(int num)`：设置 CACHED 模式每次扩容的线程数；
- `setQueueMaxCapacity(int capacity)`：动态调整关联任务队列的最大容量；
- `changeMode(TaskQueue::PriorityMode mode)`：切换关联队列的优先级调度模式。

**状态查询接口：**
- `getLiveNum()`：获取当前存活线程数；
- `getBusyNum()`：获取当前正在执行任务的忙线程数；
- `getQueueCurrentSize()`：获取任务队列当前任务数。

### 工作模式详解

#### FIXED 模式

- **核心特性**：线程数固定，(默认为 cpu 线程数) 核心线程数 = 最大线程数，无动态扩缩容；
- **守护者线程**：内置守护者线程，实时监控工作线程状态，若线程异常退出，自动清理无效线程并补充新线程，确保线程数稳定；
- **适用场景**：任务量稳定、任务执行时间均衡的场景（如后台定时任务、数据同步任务）。

#### CACHED 模式

- **核心特性**：支持动态扩缩容，线程数在核心线程数（min）和最大线程数（max）之间波动；
- **管理者线程**：定期（3 秒）监控线程池状态，执行扩缩容和无效线程清理；
- **扩容逻辑**：当队列任务积压（任务数 > 存活线程数）且存活线程数 < 最大线程数时，自动扩容（每次扩容步长可配置）；
- **缩容逻辑**：当忙线程数的 2 倍 <存活线程数（线程空闲过多）且存活线程数> 核心线程数时，自动缩容到核心线程数；
- **适用场景**：任务量波动大、任务执行时间短的场景（如 Web 服务器请求处理、临时批量任务）。

## 9. 注意事项

- 任务队列关闭后，无法提交新任务，仅能处理已入队的剩余任务；
- 任务函数中若存在未捕获的异常，会被线程池捕获并通过 `std::future` 传递给用户，建议用户在获取任务结果时通过 try-catch 捕获异常；
- 多线程池共享任务队列时，所有线程池都会响应任务入队事件，实现任务负载均衡；
- 编译时需指定 C++20 及以上标准（若无法使用 C++20, 可自行添加信号量类替代 C++20 的信号量类），并链接线程库（如 GCC 需添加 -lpthread 参数）；
- 超时检测为软超时机制，仅标记超时状态，不强行终止任务执行，避免内存泄漏或程序崩溃。

## 10. 扩展方向

本项目可基于现有架构进一步扩展，实现架构演进：
- **支持基于时间轮的任务调度**：实现延迟任务和定时任务功能
- **引入 Work-Stealing 机制**：提升多线程池调度效率，减少线程空闲
- **无锁队列（Lock-Free Queue）**：优化高并发场景下的任务入队性能
- **分布式任务调度**：结合 Kafka / Redis Stream 实现跨节点任务调度
- **添加可视化监控面板**：将性能指标输出为 JSON/CSV 或对接 Prometheus/Grafana

## 11. 开发过程中出现的问题

1. **CACHED 模式线程管理优化**
   - **问题**：核心线程容器（vector）容量持续增长，无法自动收缩
   - **原因**：频繁的线程扩缩容操作导致容器仅执行添加操作，未对已退出线程进行清理，长期运行会造成资源浪费
   - **解决方案**：设计 `WorkerStatus` 封装类，通过 `isFinish` 状态标记线程生命周期，在管理者线程中实现自动清理机制，确保容器大小始终不超过最大线程数配置

2. **并发计数一致性问题**
   - **问题**：CACHED 模式下线程繁忙计数（busyNum）断言失败
   - **原因**：尽管使用原子变量，在异常处理或提前返回等复杂控制流中，仍存在计数不一致风险，属于典型的"非原子复合操作"导致的逻辑安全问题
   - **解决方案**：采用 RAII 设计模式，将繁忙计数的增减封装到局部对象中，通过构造函数自增、析构函数自减，确保计数与线程生命周期严格绑定

3. **线程池优雅关闭机制**
   - **问题**：FIXED 模式下线程池析构时出现无限阻塞
   - **原因**：线程间信号传递不同步，部分工作线程未能及时响应关机信号
   - **解决方案**：实现双重唤醒机制，间隔 100ms 重复发送唤醒信号，确保所有线程均能及时响应并退出

4. **任务队列与线程池解耦设计**
   - **问题**：线程池析构过程中出现无限阻塞
   - **原因**：析构顺序不当，先解绑任务队列后无法唤醒阻塞中的工作线程，导致线程无法正常退出
   - **解决方案**：优化析构流程，先发送关机信号，唤醒所有工作线程，待线程响应后再执行任务队列解绑操作

5. **FIXED 模式线程自愈机制**
   - **问题**：FIXED 模式下核心线程异常退出后，线程池逐渐失去处理能力
   - **原因**：缺乏线程状态监控机制，核心线程异常退出后无法自动补充，导致线程池逐渐退化
   - **解决方案**：基于 Semaphore（C++20 特性，或使用互斥锁与条件变量实现）实现守护者线程，实时监控线程状态，自动补充异常退出的核心线程

6. **线程池重启一致性保证**
   - **问题**：FIXED 模式下线程池重启后实际线程数与配置不一致
   - **原因**：守护者线程关闭时采用 break 退出阻塞，未正确释放信号量资源，导致重启时守护者线程误判线程数量
   - **解决方案**：为守护者线程添加线程数校验逻辑，仅当实际线程数小于配置值时才创建新线程

7. **编译性能优化**
   - **问题**：Visual Studio 编译器报"堆内存不足"错误
   - **原因**：所有代码集中在单个 .hpp 头文件中，包含大量模板定义、类型推导和复杂嵌套结构，导致编译器处理时内存消耗过大
   - **解决方案**：采用分离编译模式，将代码拆分为 `ThreadPool.h`（包含声明、模板和结构体定义）和 `ThreadPool.cpp`（包含非模板实现），显著降低编译器内存占用并提升编译速度
